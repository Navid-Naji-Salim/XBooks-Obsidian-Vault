# The Question: "There are files/code in the project that are not mentioned in the roadmap you gave me. Why is that? Should i not literally read everything?"

Not needed at first, no.

The roadmap was meant to teach the system efficiently, not exhaustively. Some files are “architecture-bearing” files: they explain how the app thinks, moves data, authenticates users, talks to the database, and renders workflows. Those are the files worth reading line by line first.

Other files are support files or generated files. You can read them later, but reading them too early gives you noise before structure.

For example:

- `backend/generated/prisma/*` is generated by Prisma. You usually don’t study it line by line unless you’re debugging Prisma types.
- `frontend/src/vite-env.d.ts` is just TypeScript environment typing.
- `frontend/index.html` matters, but it is tiny and not where the app logic lives.
- `icons.tsx` matters visually, but it won’t teach you much about architecture.
- config files are important, but they make more sense after you understand the app flow.

Eventually, yes, if your goal is mastery, read everything. But the order matters. First understand the spine of the system. Then read the ribs. Then read the small connective tissue.

A good rule:

1. Read files that define the domain.
2. Read files that start the app.
3. Read files that route requests/pages.
4. Read files that transform and validate data.
5. Read feature files.
6. Read shared UI/helpers.
7. Read config.
8. Read generated code only when you have a reason.

So no, skipping a file in the first roadmap does not mean it is useless. It means it is not the best next file for understanding the system.
